# Respan

Respan is the active product name for the platform that was previously searched under Keywords AI. It sits in the LLMOps lane: observe model calls, evaluate prompts and models, manage prompt versions, monitor cost and latency, and route traffic through an OpenAI-compatible gateway.

The buyer reason to care is consolidation. If traces live in one tool, evals in another, prompts in a spreadsheet, and routing in app code, Respan can become the shared surface for AI product quality and cost control.

## System Verdict

> **Pick Respan when production AI behavior needs one control room.** It is strongest when a team needs traces, metrics, evals, prompt management, monitors, spend limits, and gateway routing tied to the same production traffic.
>
> **Skip it when code-first evals are enough.** [DeepEval](/tools/deepeval/), [Ragas](/tools/ragas/), [promptfoo](/tools/promptfoo/), or [DSPy](/tools/dspy/) fit better when developers only need local tests, notebooks, or CI checks.
>
> **Best plan guidance:** start on Free to validate instrumentation and workflow fit. Move to Team after shared product workflows justify collaboration and seat costs. Use Enterprise for support, compliance, HIPAA BAA, or custom terms.

## When To Pick Respan

- **You need traces and evals together.** Respan is a better fit when production logs should turn into datasets, experiments, monitors, and prompt improvements.
- **You need a gateway with evidence.** The gateway routes calls while logging latency, tokens, fallback behavior, cache status, and metadata.
- **You need prompt release discipline.** Prompt versions and experiments help teams compare changes before pushing them into production.
- **You need spend and reliability controls.** Monitors, limits, and model fallback are useful when model calls become a shared platform concern.
- **You want a managed platform.** Respan is easier to adopt than assembling a gateway, traces, eval dashboards, and prompt registry from separate components.

## When To Pick Something Else

- **Open-source gateway:** [LiteLLM](/tools/litellm/) when deployment control and self-hosted routing matter more than a managed LLMOps UI.
- **Gateway governance:** [Portkey](/tools/portkey/) when API-key policy, guardrails, routing, and gateway governance are the main purchase.
- **Hosted eval operations:** [Braintrust](/tools/braintrust/) or [LangSmith](/tools/langsmith/) when datasets, traces, prompt testing, and release evidence are the core team workflow.
- **Open-source observability:** [Arize Phoenix](/tools/arize-phoenix/) or [LangWatch](/tools/langwatch/) when self-hosting and OpenTelemetry alignment matter.
- **Code-first evals:** [DeepEval](/tools/deepeval/), [Ragas](/tools/ragas/), or [promptfoo](/tools/promptfoo/) when tests should live close to code first.

## Failure Modes

- **Pricing needs checkout confirmation.** The June 28 check verified Free, Team, Enterprise, and an additional-seat line, but not a clean Team base price.
- **Gateway latency matters.** Routing through a new data plane can affect production latency.
- **Eval quality depends on datasets.** Weak datasets and weak evaluators can make prompt changes look safer than they are.
- **Provider bills remain separate.** Respan helps see and control usage, but it does not make model calls free.
- **Data policy needs review.** Prompts, outputs, metadata, and traces can contain sensitive customer or product information.
